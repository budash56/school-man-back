Below is a simple, router-agnostic step-by-step you can follow at home, plus specific official guides you can open. If you share your router brand/model later, I can tailor it to your exact menus.

Step-by-step (router + DNS)
1) Find your router’s IP and log in
   - On Windows, open Command Prompt and run ipconfig, then look for Default Gateway.
   - On macOS, open Terminal and run route -n get default.
   - Open that IP in your browser and log in.

2) Reserve a static IP via DHCP (recommended)
   - Look for DHCP, LAN, or Address Reservation in your router’s admin UI.
   - Find your server in the DHCP client list, then reserve an IP for it.
   - This makes the router always assign the same IP to that device.

3) Apply/Save and renew the server’s IP
   - Save the reservation.
   - Reboot the server or renew DHCP, then confirm it got the reserved IP.

4) (Optional) Add a local DNS name
   - If your router supports local DNS/host entries, add a hostname like schoolman.lan -> 192.168.1.50.
   - This allows devices on your LAN to reach the server by name, not just IP.

5) Test from another device
   - On another device in the same Wi-Fi/LAN, try:
     - ping 192.168.1.50
     - nslookup schoolman.lan (if local DNS is set)
     - Open the app in a browser: http://192.168.1.50:5173 or http://schoolman.lan:5173

If your router doesn’t support local DNS
Use one of these simple options:
1) Hosts file on each client (Windows/Mac/Linux)
2) Local DNS appliance (Pi-hole or AdGuard Home) running on the same server

Docker, GitHub, and deployment strategy

Use both GitHub and Docker. They solve different problems.

Short version
- GitHub: source control, version history, collaboration, backup.
- Docker: reproducible runtime for backend, frontend, and PostgreSQL.
- Best approach: keep code in GitHub, deploy with Docker Compose on each local server.

Why Docker helps here
- Same environment on every machine.
- No manual Node/Postgres/version setup per server.
- Easier upgrades and rollback.
- Backend, frontend, and DB can start together.
- Good fit for LAN-only/intranet deployments.

For this system
You likely want 3 containers:
1) frontend
   - built React/Vite app served by Nginx.
2) backend
   - NestJS API.
3) postgres
   - database with a persistent volume.

Optional later
4) pgadmin or another DB admin tool.
5) reverse proxy like nginx or traefik.

How GitHub fits
- Store the code there.
- Each server clones the repo or pulls updates.
- Then runs docker compose up -d --build.

So GitHub distributes the code, Docker standardizes execution.

Best deployment model for intranet
- One machine inside the school LAN acts as the server.
- Docker Compose runs the stack.
- Other machines connect by LAN IP or local DNS name.
- Internet is only needed from the backend container for Gmail SMTP.

Recommended practical setup
- docker-compose.yml
- Dockerfile for backend
- Dockerfile for frontend
- named Docker volume for Postgres data
- .env file per server for:
  - DB credentials
  - JWT secrets
  - email SMTP config
  - API base URLs

How to deploy to multiple local machines
Two reasonable options:

1) Simple/manual
- Install Docker on each machine.
- Clone repo from GitHub.
- Copy .env.
- Run docker compose up -d.
- Best for a few servers.

2) Image-based
- Build Docker images once.
- Push images to GitHub Container Registry or Docker Hub.
- Each machine only pulls images and runs Compose.
- Better if many servers or weak machines.

For this scale, start with option 1, then move to option 2 later if needed.

Database handling
Use Docker for Postgres too, but:
- mount a persistent volume.
- back it up regularly with pg_dump.
- never rely only on the container itself.

Docker makes DB setup easy, but backups are still your responsibility.

Good pattern
- GitHub for source.
- Docker Compose for deployment.
- Postgres volume for data.
- periodic SQL backups to a host folder or external disk.

What makes this easy across machines
- one Compose file.
- one .env.example.
- one deployment doc.
- one command to start:
  docker compose up -d --build

Local intranet-specific advice
- Use fixed LAN IP or local DNS hostname.
- Keep ports internal only.
- No need to expose to the public internet.
- Email still works as long as that server has outbound internet.

What I would recommend for SchoolMan
1) Put both repos in GitHub.
2) Add Dockerfiles to front and back.
3) Add one docker-compose.yml for:
   - frontend
   - backend
   - postgres
4) Add persistent DB volume.
5) Add backup script.
6) Deploy on one school server first.
7) Reuse the same stack on any replacement machine.

If you want maximum simplicity
You can also create one separate repo like school-man-deploy.
That repo would only contain:
- docker-compose.yml
- deployment docs
- reverse proxy config
- .env.example

Then both app repos remain clean, and deployment is centralized.

Bottom line
- Do not choose GitHub or Docker.
- Use GitHub + Docker.
- That is the cleanest way to deploy the same intranet system on multiple local machines with minimal setup differences.
